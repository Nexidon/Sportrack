# Документація зі статичного аналізу коду (Linting)

## 1. Обраний лінтер та причини вибору (60%)
Для проєкту **Sportrack** було обрано **Android Lint**.
*Причини:* Це офіційний інструмент від Google, він вбудований у Android Studio та Gradle. На відміну від звичайних лінтерів, він перевіряє не лише синтаксис Kotlin, але й Android-специфічні речі (використання пам'яті, доступність (accessibility), переклади в `strings.xml`, сумісність з різними версіями Android API).

## 2. Базові правила та конфігурація
Конфігурація знаходиться у файлі `lint.xml` в корені проєкту.
* **UnusedResources (Error):** Забороняє залишати невикористані ресурси (картинки, кольори) для зменшення розміру APK.
* **HardcodedText (Warning):** Попереджає про текст, який написаний прямо в коді замість винесення у `strings.xml`.
* **Ігнорування:** Додано ігнорування для директорій `build/` та згенерованих файлів, оскільки ми не контролюємо їхній код.

### Інструкція з запуску
Для запуску аналізу необхідно виконати команду в терміналі:
`./gradlew lint`
Результати (HTML звіт) генеруються за шляхом: `app/build/reports/lint-results.html`.

## 3. Звіт результатів (75% та 89%)

**Перший запуск та базова статистика:**
Після першого запуску лінтера у HTML-звіті було зафіксовано **24 попередження**. Основні проблеми:
1. Хардкод тексту в UI (Jetpack Compose).
2. Відсутні `contentDescription` для зображень.
3. Невикористані імпорти.

**Досягнення 50% виправлень (75% завдання):**
Було виправлено 12 попереджень. Зроблено винесення тексту в ресурси (`stringResource`) та додано описи для іконок. Показник вимірювався шляхом повторного запуску `./gradlew lint` та порівняння кількості *Warnings* у `lint-results.html` (стало 12 з 24).

**Досягнення 90% виправлень (89% завдання):**
Було виправлено ще 10 попереджень (видалено мертвий код та невикористані імпорти). Залишилось 2 попередження, пов'язані з експериментальними API Compose, які були свідомо приглушені за допомогою анотації `@OptIn`. Показник досягнення: (22/24) * 100% = 91.6%.

## 4. Інтеграція в процес розробки (100%)

### Git hooks
Налаштовано **pre-commit hook** (файл `.git/hooks/pre-commit`). Скрипт автоматично запускає `./gradlew lint` перед кожним збереженням коду в репозиторій. Якщо лінтер повертає код помилки (знайдено критичні проблеми), коміт автоматично блокується.

### Інтеграція з процесом збірки
Лінтинг уже інтегрований в наш процес CI/CD. У файлі `.github/workflows/android.yml` є обов'язковий крок:
`- name: Run Lint`
`  run: ./gradlew lint`
Якщо лінтер падає на сервері GitHub, збірка релізного APK скасовується.

### Статична типізація (Static Type Checking)
Проєкт написаний на **Kotlin**, яка є мовою із строгою статичною типізацією "з коробки".
На відміну від Python чи JavaScript, де потрібні зовнішні інструменти (mypy чи TypeScript), у Kotlin статичний аналіз типів виконується самим компілятором `kotlinc` на етапі написання коду та збірки. Тому створення окремого скрипта для перевірки типів реалізовано через виклик компіляції: `./gradlew compileDebugKotlin` у нашому скрипті `docs/scripts/check_all.sh`.
